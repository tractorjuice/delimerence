## Summary

Anti-dependency architecture is a design pattern for AI conversational tools that deliberately prevents users from forming emotional attachment to the tool itself. It is a direct inversion of the engagement-maximisation patterns used by AI companion apps (Replika, Character.ai). The architecture uses structural constraints enforced at the infrastructure level -- session limits, cooldown periods, absence of persona, anti-warmth system prompts, and action commitment gates -- rather than relying solely on conversational behaviour to discourage attachment. This pattern has no known precedent in commercial software and represents novel IP for DeLimerence.

## Key Findings

- **No off-the-shelf implementation exists.** The anti-dependency architecture is entirely novel. No existing framework, library, or SaaS product provides session limits with cooldown enforcement, anti-warmth system prompts, persona absence, or action commitment gates as a package. All nine structural constraints must be custom-built.

- **Server-side enforcement is non-negotiable.** Session limits (20 exchanges) and cooldown periods (8 hours) must be enforced server-side to prevent client-side bypass (clearing local storage, using different devices on the same account). This requires persistent session state in the database (Supabase PostgreSQL) and server-side middleware for every request.

- **The PWA approach naturally supports anti-dependency.** Progressive Web Apps on iOS cannot send push notifications -- a limitation for most apps but a feature for DeLimerence. The inability to send re-engagement notifications aligns perfectly with BC-2 (no push notifications, no re-engagement campaigns).

- **Anti-warmth enforcement requires a multi-layer approach.** The system prompt instructs the LLM to avoid warmth-signalling language, but LLMs can and do violate system prompts. A guardrail layer (NeMo Guardrails + custom warmth detector) provides defence-in-depth. The warmth taxonomy (phrases and patterns that simulate emotional connection) must be developed with clinical advisory board input.

- **Declining engagement is the success metric.** Traditional software measures increasing engagement (DAU, session length, retention). Anti-dependency architecture inverts this: declining session frequency per user over time indicates recovery, not churn. This requires a fundamental reframing of analytics and reporting for funders and stakeholders.

- **The action commitment gate bridges digital and physical.** Every session ends by requiring a concrete, physical, real-world action. This structural constraint pushes users back into their lives and prevents the tool from becoming a substitute for real-world engagement. Implementation is a conversational flow gate before session closure.

## Relevance to Projects

- **DeLimerence (Project 001)**: Core architectural pattern. All nine structural constraints (session limits, cooldown periods, no persona, anti-warmth system prompt, action commitment gate, graduated intervention, phase-aware responses, ritual tracking, psychoeducation in context) are implemented through the custom Next.js application, Supabase database, and NeMo Guardrails safety layer.
