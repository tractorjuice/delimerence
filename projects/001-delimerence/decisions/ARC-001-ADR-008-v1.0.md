## Context

DeLimerence is a 24/7 limerence recovery tool designed for psychologically vulnerable users who may need access at any time of day, including during acute crisis episodes at unsociable hours. The application must be accessible via both web and mobile devices, maintain 99.9% uptime (NFR-A-001), enforce UK data residency for all user data (TC-1), and scale horizontally from 5,000 users at launch to 100,000 users over a 3-year growth trajectory (NFR-S-001).

The deployment infrastructure decision is architecturally significant because it directly affects availability (a user in crisis at 3am must be able to reach the tool — R-007), data residency compliance (all processing must occur within the UK — TC-1), and the anti-dependency architecture. Specifically, the choice of client platform has a material impact on behavioural constraint BC-2 (no push notifications or re-engagement): a platform that structurally prevents push notifications is preferable to one that merely promises not to use them.

Research (ARC-001-RSCH-v1.0) evaluated application framework options and concluded that a Next.js Progressive Web App deployed on AWS ECS Fargate in eu-west-2 (London) is the recommended approach. The PWA model is uniquely aligned with the anti-dependency architecture — iOS PWAs cannot send push notifications, which is normally a limitation but for DeLimerence is a structural guarantee that the no-re-engagement constraint cannot be circumvented by future product decisions. The research also confirmed that AWS Fargate provides serverless container orchestration with 99.99% Multi-AZ SLA, exceeding the 99.9% availability target, with Bedrock integration in the same region for minimal LLM call latency (ADR-001).

This decision addresses NFR-A-001 (99.9% Availability), NFR-A-002 (Disaster Recovery — RPO 1hr, RTO 4hrs), NFR-A-003 (Graceful Degradation), NFR-S-001 (Horizontal Scaling), NFR-P-001 (Response Time < 3s page load), NFR-P-002 (Throughput — 10K sessions/day Year 1), TC-1 (UK Data Residency), and BC-2 (No push notifications). It directly mitigates R-007 (Crisis not detected — high availability ensures the tool is accessible when users need it most).

## Decision Drivers

- **24/7 crisis accessibility**: Users in acute limerent distress may need the tool at 3am; downtime is not a UX inconvenience but a safety risk [R-007, NFR-A-001]
- **UK data residency mandate**: All user data must be stored and processed within the United Kingdom; health-related data for vulnerable users demands unambiguous jurisdictional compliance [TC-1, NFR-C-001]
- **Anti-dependency structural enforcement**: The platform choice should make re-engagement impossible by design, not merely by policy — reducing the risk that future product owners reintroduce engagement dark patterns [BC-2]
- **Horizontal scaling requirement**: Infrastructure must scale from 5,000 users (Year 1) to 100,000 users (Year 3) without architectural rework [NFR-S-001]
- **Bedrock co-location**: The LLM provider (ADR-001) runs on AWS Bedrock in eu-west-2; deploying the application in the same region minimises latency and keeps all data within UK boundaries [NFR-P-001]
- **Cost efficiency at startup scale**: Infrastructure costs must be proportionate to a startup-stage project with limited funding [BC-1]
- **Operational simplicity**: Small development team requires managed infrastructure with minimal server administration overhead
- **Time to market**: Q4 2026 public beta target (BC-1) constrains the development approach — single codebase preferred over multi-platform native development

## Considered Options

1. **Do Nothing** — No deployment infrastructure
2. **Next.js PWA on AWS Fargate eu-west-2** (chosen)
3. **Vercel + Edge Functions**
4. **Native Mobile Apps (iOS + Android)**

## Decision Outcome

**Chosen option: "Next.js PWA on AWS Fargate eu-west-2"**, because it is the only option that simultaneously guarantees UK data residency, provides serverless auto-scaling aligned with the 5K-to-100K growth trajectory, co-locates with Bedrock in the same region for minimal LLM latency, and structurally enforces the no-push-notification anti-dependency constraint through PWA platform limitations rather than policy alone.

### Y-Statement

In the context of deploying a 24/7 limerence recovery tool with UK data residency requirements, facing the need for auto-scaling infrastructure that naturally supports anti-dependency constraints, we decided for Next.js PWA on AWS Fargate in eu-west-2, to achieve serverless scalability with guaranteed UK data residency and natural push notification prevention, accepting AWS vendor lock-in and container management complexity.

### Consequences

**Positive:**

- UK data residency is guaranteed at infrastructure level — eu-west-2 region ensures all compute and data remain on UK soil, satisfying TC-1 and simplifying ICO/DPIA compliance
- PWA structurally prevents iOS push notifications — the anti-dependency constraint BC-2 is enforced by platform limitation, not by developer discipline or product policy
- Serverless containers eliminate server management — the small development team can focus on application code rather than infrastructure operations
- Auto-scaling is built in — Fargate scales tasks based on CPU/memory thresholds, directly addressing NFR-S-001 without manual capacity planning
- Bedrock co-location in eu-west-2 minimises LLM call latency and ensures LLM data never leaves the UK
- Single codebase (Next.js) serves both web and mobile users, reducing development cost by approximately 40% compared to native mobile alternatives
- No app store submission required — avoids Apple and Google health app review processes, which add regulatory friction and timeline risk
- Estimated infrastructure cost of GBP 500-1,500/month is proportionate to startup-stage funding

**Negative:**

- AWS vendor lock-in — migrating away from Fargate, ALB, and CloudFront would require significant re-engineering of deployment infrastructure
- Fargate cold starts may cause latency spikes for the first request after a scale-up event; mitigated by maintaining a minimum task count of 2 for always-on availability
- More operationally complex than PaaS alternatives (Vercel) — requires configuring ALB, CloudFront, auto-scaling policies, and container images
- PWA on iOS has limited capabilities compared to native apps (no background processing, limited storage) — acceptable given the clinical, minimal UX requirement (NFR-U-001)
- Container image management adds a build/deploy step not present in pure serverless (Lambda) or PaaS approaches

### Confirmation

The decision will be validated through:

- Load testing to confirm auto-scaling meets NFR-S-001 (5K to 100K concurrent user scaling) with response times under 3 seconds (NFR-P-001)
- Availability monitoring against 99.9% uptime target (NFR-A-001) with alerting for any downtime exceeding 8.76 hours per year
- Disaster recovery drill validating RPO 1 hour and RTO 4 hours (NFR-A-002) using AWS cross-region failover
- Verification that PWA installed on iOS cannot send push notifications (BC-2 structural enforcement)
- Latency measurement confirming Fargate-to-Bedrock round-trip within eu-west-2 stays under acceptable thresholds for NFR-P-001
- Infrastructure cost monitoring against GBP 500-1,500/month estimate, with cost alerts at 80% threshold
- Graceful degradation testing: confirm static fallback content is served when LLM or database is unavailable (NFR-A-003)

## Pros and Cons of the Options

### Option 1: Do Nothing — No Deployment Infrastructure

**Description**: Do not provision any deployment infrastructure. The application has no hosting environment and cannot be delivered to users.

- Good, because zero infrastructure cost
- Good, because no operational complexity
- Bad, because the product cannot be delivered — the core mission of DeLimerence is unachievable
- Bad, because all requirements (NFR-A-001, NFR-S-001, NFR-P-001, TC-1) remain unaddressed
- Bad, because R-007 (crisis not detected) becomes certain — the tool does not exist for users to access
- Bad, because project objectives G-1 through G-5 cannot be met

### Option 2: Next.js PWA on AWS Fargate eu-west-2 (Chosen)

**Description**: Containerised Next.js application deployed on AWS ECS Fargate (serverless containers) in the London region (eu-west-2). Application Load Balancer (ALB) for traffic distribution, CloudFront CDN for static asset delivery, auto-scaling based on CPU/memory thresholds. Progressive Web App (PWA) manifest enables "Add to Home Screen" on mobile devices for app-like experience without app store distribution. Integrates with Bedrock (ADR-001) in the same region and Supabase (ADR-005/006) as the external database layer. Estimated infrastructure cost of GBP 500-1,500/month at Year 1 scale.

- Good, because UK data residency is guaranteed at infrastructure level — eu-west-2 ensures all compute occurs on UK soil [TC-1]
- Good, because Fargate is serverless — no EC2 instances to patch, scale, or maintain; development team focuses on application code
- Good, because auto-scaling is built in — Fargate scales tasks horizontally based on CPU and memory thresholds [NFR-S-001]
- Good, because 99.99% Multi-AZ SLA exceeds the 99.9% availability target [NFR-A-001]
- Good, because Bedrock integration in the same region minimises LLM call latency and avoids cross-border data transfer [ADR-001, NFR-P-001]
- Good, because PWA naturally prevents push notifications on iOS — anti-dependency constraint BC-2 is enforced by platform limitation, not developer discipline
- Good, because single codebase serves web and mobile — approximately 40% cost reduction compared to native development [ARC-001-RSCH-v1.0]
- Good, because no app store approval process — avoids health app review scrutiny from Apple and Google
- Good, because CloudFront CDN improves page load performance for static assets [NFR-P-001]
- Good, because cost-effective at startup scale — GBP 500-1,500/month infrastructure cost is proportionate to early-stage funding
- Neutral, because Fargate cold starts add latency on scale-up events — mitigated by minimum task count of 2
- Bad, because AWS vendor lock-in — Fargate, ALB, CloudFront, and CloudWatch create significant switching cost
- Bad, because more operationally complex than PaaS (Vercel) — requires container image management, ALB configuration, auto-scaling policy tuning
- Bad, because PWA on iOS has limited capabilities (no background processing, limited local storage) — acceptable given clinical minimal UX requirement [NFR-U-001]

### Option 3: Vercel + Edge Functions

**Description**: Deploy the Next.js application on Vercel's global edge network with edge functions for server-side rendering and API routes. Automatic scaling, global CDN, and native Next.js integration (Vercel is the creator of Next.js). Serverless functions execute at the edge location nearest to the user.

- Good, because simplest deployment workflow — git push triggers automatic builds and deployments
- Good, because excellent Next.js integration — Vercel is the framework creator and optimises for Next.js features
- Good, because automatic scaling with no configuration required
- Good, because global CDN provides fast page loads worldwide
- Good, because zero infrastructure management overhead
- Bad, because no guaranteed UK-only data processing — edge functions execute at the nearest edge location globally, which may be outside the UK [TC-1 violation]
- Bad, because Vercel's core infrastructure is US-based — user data transits through US infrastructure for builds, deployments, and serverless function cold starts
- Bad, because limited control over data residency — Vercel does not offer region-locked deployments that guarantee all processing stays in the UK
- Bad, because less suitable for health data compliance — Vercel lacks NHS DSPT, healthcare-specific compliance certifications, and UK-focused data processing agreements
- Bad, because limited observability — Vercel's monitoring is less comprehensive than CloudWatch + Grafana for production health applications
- Bad, because Bedrock integration requires cross-cloud networking — LLM calls from Vercel edge functions to AWS eu-west-2 add latency and complexity
- Bad, because vendor lock-in to Vercel-specific features (edge middleware, ISR caching) with limited portability

### Option 4: Native Mobile Apps (iOS + Android)

**Description**: Build native mobile applications using React Native or Flutter, providing full platform capabilities on both iOS and Android. Separate web application would also be required for desktop access. Distributed via Apple App Store and Google Play Store.

- Good, because full access to native platform capabilities (camera, biometrics, local storage, background processing)
- Good, because native push notification capability (though it would be disabled per BC-2)
- Good, because app store distribution provides a familiar installation experience for users
- Good, because native performance and animations for mobile UI
- Bad, because 3-4x development cost compared to PWA — React Native/Flutter requires platform-specific code, testing, and maintenance [ARC-001-RSCH-v1.0: 67% more expensive]
- Bad, because longer time to market — jeopardises Q4 2026 public beta target [BC-1]
- Bad, because push notification capability exists even if disabled — creates temptation for future product owners to enable re-engagement, undermining BC-2 structural guarantee
- Bad, because app store review process adds regulatory risk — Apple and Google scrutinise health apps, adding timeline uncertainty and potential rejection
- Bad, because app store updates require review cycles — critical safety updates cannot be deployed instantly
- Bad, because requires separate web application for desktop users — increases codebase maintenance burden
- Bad, because app store fees (Apple 30% revenue share) add cost if the product ever monetises
