## Status

**DRAFT**

## Context

DeLimerence is an anti-dependency conversational AI tool that helps users recognise and recover from limerence using CBT, ERP, and psychoeducation. The tool processes special category health data as defined by UK GDPR Article 9: session transcripts containing intimate mental health disclosures, ritual tracking records documenting compulsive behaviours, phase assessments indicating psychological state, and self-assessment scores measuring distress intensity. This data is collected from psychologically vulnerable users with compromised executive function [LM12-C3].

All data must be stored and processed within the United Kingdom (TC-1). Field-level encryption is required for sensitive content such as session transcripts and ritual tracking data (NFR-SEC-003). The data retention policy mandates that session transcripts are deleted after 12 months, ritual tracking summary data is retained while the account is active, and all data is deleted within 30 days of account deletion.

A data breach of this material would expose intimate mental health disclosures from a uniquely vulnerable population, triggering ICO enforcement action (fines up to GBP 17.5M), complete loss of user trust, and potential withdrawal of clinical advisory board endorsement [R-003]. The LLM provider's data processing agreement must also prevent storage of session content, adding a secondary compliance obligation [R-010].

The data model comprises four core entities: User Profile (authentication and consent), Session Data (~500,000 sessions/year at Year 2 scale), Ritual Tracking (~2.5M records/year at Year 2 scale), and Self-Assessment Scores. Total data volume is projected at 10TB over 3 years (NFR-S-002). Ritual tracking has strong relational patterns — many-to-one relationships with both User Profile and Session Data — which favour a relational database over document stores.

This decision addresses DR-001 (User Profile), DR-002 (Session Data), DR-003 (Ritual Tracking Data), DR-004 (Self-Assessment Scores), NFR-SEC-003 (Data Encryption), NFR-C-001 (UK GDPR Compliance), NFR-S-002 (Data Volume Scaling), and TC-1 (UK data residency). It directly mitigates R-003 (UK GDPR special category data breach, residual 12) and R-010 (LLM provider DPA inadequate, residual 6).

## Decision Drivers

- **Special category data sensitivity**: Session transcripts and ritual tracking records constitute Article 9 health data from users with compromised executive function — breach consequences are severe and amplified by user vulnerability [R-003]
- **UK data residency mandate**: All user data must be stored and processed within the UK (TC-1); cross-border transfer is not acceptable for this data classification
- **Per-user data isolation**: Role-based access control requires strict isolation between users — no user may access another user's session data, ritual tracking, or self-assessment scores [NFR-SEC-002]
- **Field-level encryption requirement**: Session transcripts and ritual tracking data require application-level encryption beyond standard encryption at rest [NFR-SEC-003]
- **Cost-effectiveness**: The project is founder-led with a GBP 275K 3-year TCO budget [ARC-001-RSCH-v1.0]; database cost must align with startup economics
- **Auth+DB co-location**: ADR-005 selected Supabase Auth; using the same Supabase instance for the database eliminates a separate procurement, reduces integration complexity, and provides a single vendor relationship
- **Relational data model**: Ritual tracking has many-to-one relationships with both User Profile and Session Data; self-assessments relate to both users and sessions — relational patterns that favour SQL over document stores
- **Data retention automation**: Session transcripts must be auto-deleted after 12 months; all user data must be purged within 30 days of account deletion — requires reliable automated retention enforcement
- **Scaling trajectory**: 10TB over 3 years with clear growth from 5,000 to 100,000 registered users [NFR-S-002]

## Considered Options

1. **Do Nothing** — No persistent data store
2. **Supabase PostgreSQL with RLS** (chosen)
3. **AWS RDS PostgreSQL (Self-Managed)**
4. **MongoDB Atlas**

## Decision Outcome

**Chosen option: "Supabase PostgreSQL with RLS"**, because it provides cost-effective managed PostgreSQL with built-in Row Level Security for per-user data isolation, field-level encryption via pgcrypto, UK hosting via AWS eu-west-2, and co-location with the Supabase Auth instance selected in ADR-005 — delivering the strongest combination of data protection, compliance, and operational simplicity within the project's budget constraints.

### Y-Statement

In the context of storing special category health data for vulnerable users, facing UK data residency requirements and the need for per-user data isolation, we decided for Supabase PostgreSQL with Row Level Security, to achieve cost-effective data persistence with built-in access control and field-level encryption, accepting vendor dependency on Supabase's managed infrastructure.

### Consequences

**Positive:**

- Combined auth and database on a single Supabase Pro instance (GBP 25/month) eliminates separate database procurement and reduces integration overhead
- Row Level Security enforces per-user data isolation at the database level without relying on application code — a user's RLS policy cannot be bypassed by application bugs
- pgcrypto extension provides field-level encryption for session transcripts and ritual tracking data, meeting NFR-SEC-003 beyond standard encryption at rest
- Supabase provides encryption at rest (AES-256) and in transit (TLS 1.3+) by default
- SOC 2 Type 2 certification and GDPR-compliant Data Processing Agreement available
- UK data residency achieved via Supabase's AWS eu-west-2 (London) hosting
- Automated hourly backups for user data with point-in-time recovery
- Open-source core (MIT/Apache 2.0) provides a genuine exit strategy — self-hosting is feasible if Supabase's managed service becomes unsuitable
- PostgreSQL's maturity and ecosystem provide strong support for the relational data model (foreign keys, constraints, indexing for the ritual tracking relationships)
- Real-time subscriptions available for live state updates (cooldown timers, session state)

**Negative:**

- Vendor dependency on Supabase's managed infrastructure — Supabase is a relatively young company (founded 2020), creating a vendor viability consideration
- Less direct control than self-managed PostgreSQL on AWS RDS — database configuration options are constrained by Supabase's managed layer
- Supabase's UK hosting relies on their AWS infrastructure partnership; any change to Supabase's hosting options could impact data residency compliance
- Pro tier provides 8GB database storage — sufficient for early stages but will require compute add-ons or tier upgrade as data volume approaches 10TB over 3 years
- No dedicated UK data centre from Supabase directly — reliance on AWS eu-west-2 availability through Supabase's infrastructure
- Data retention automation (12-month transcript deletion, 30-day account purge) must be implemented at the application layer using PostgreSQL scheduled jobs or Supabase Edge Functions

### Confirmation

The decision will be validated through:

- Verification that Supabase Pro instance is deployed in AWS eu-west-2 (London) with data residency confirmed in the Supabase project settings
- RLS policies tested to confirm per-user data isolation — no user can query another user's session data, ritual tracking, or self-assessment scores
- pgcrypto field-level encryption verified for session transcript and ritual tracking content fields
- Data retention automation tested: session transcripts confirmed deleted after 12 months; all user data confirmed purged within 30 days of account deletion
- Penetration testing confirms RLS policies cannot be bypassed via SQL injection or API manipulation [ARC-001-RISK-v1.0, Action 11]
- DPIA submission references Supabase's SOC 2 Type 2 certification and DPA as evidence of adequate data processor controls [ARC-001-RISK-v1.0, Action 5]
- Data volume monitoring confirms storage scaling plan is adequate as user base grows toward Year 2/Year 3 projections

## Pros and Cons of the Options

### Option 1: Do Nothing — No Persistent Data Store

**Description**: Do not implement a persistent data store. All data exists only in-memory during the active session and is discarded when the session ends.

- Good, because zero data breach risk — no data at rest to compromise
- Good, because no GDPR data subject rights obligations for stored data
- Bad, because no session persistence — users cannot resume or reference previous conversations
- Bad, because no ritual tracking — the core therapeutic mechanism (tracking slip-ups and resistance over time) cannot function
- Bad, because no cooldown enforcement — anti-dependency session limits cannot be enforced across sessions without persistent state [ADR-002]
- Bad, because no self-assessment tracking — longitudinal outcome measurement is impossible
- Bad, because no research data collection — anonymised outcome data for clinical validation cannot be gathered
- Bad, because the product cannot function as designed — fails to meet DR-001, DR-002, DR-003, DR-004, FR-002, FR-004, FR-005

### Option 2: Supabase PostgreSQL with RLS (Chosen)

**Description**: Supabase Pro tier (GBP 25/month) provides managed PostgreSQL with Row Level Security for per-user data isolation, encryption at rest (AES-256), encryption in transit (TLS 1.3+), and UK hosting via AWS eu-west-2. Bundled with Supabase Auth (ADR-005) on the same instance. Field-level encryption for session transcripts and ritual data implemented via pgcrypto extension. Automated hourly backups with point-in-time recovery. SOC 2 Type 2 certified, GDPR-compliant DPA available.

- Good, because combined auth and database on a single instance reduces procurement, integration, and operational complexity
- Good, because RLS provides per-user data isolation enforced at the database level — application code bugs cannot bypass data isolation
- Good, because pgcrypto enables field-level encryption for sensitive content (session transcripts, ritual data) beyond standard encryption at rest
- Good, because SOC 2 Type 2 certified with GDPR-compliant Data Processing Agreement
- Good, because cost-effective at GBP 25/month (Pro tier) — aligned with startup budget constraints
- Good, because UK data residency via AWS eu-west-2 (London)
- Good, because open-source core provides genuine exit strategy via self-hosting
- Good, because PostgreSQL maturity provides strong relational data model support for the ritual tracking entity relationships
- Neutral, because Pro tier 8GB storage is adequate for Year 1 but will require scaling as data volume grows toward 10TB
- Bad, because vendor dependency on a relatively young company (Supabase, founded 2020)
- Bad, because less control than self-managed PostgreSQL — configuration constrained by Supabase's managed layer
- Bad, because UK hosting relies on Supabase's AWS infrastructure partnership, adding an indirect dependency

### Option 3: AWS RDS PostgreSQL (Self-Managed)

**Description**: AWS RDS PostgreSQL deployed in eu-west-2 (London) with self-managed configuration, security, and operations. Direct AWS relationship with full control over database configuration, networking, and encryption. Estimated cost GBP 150-300/month for comparable capability.

- Good, because direct AWS control with full configuration flexibility
- Good, because mature managed service with extensive enterprise features (Multi-AZ, read replicas, automated patching)
- Good, because UK data residency guaranteed via AWS eu-west-2 with direct contractual relationship
- Good, because proven scaling path to 10TB+ with well-documented AWS RDS limits and upgrade procedures
- Good, because independent of Supabase vendor viability — AWS is a Tier 1 cloud provider
- Bad, because significantly more expensive at GBP 150-300/month (6-12x the Supabase Pro cost)
- Bad, because requires separate authentication solution — Supabase Auth (ADR-005) would need a different database connection or a separate auth provider entirely
- Bad, because RLS must be configured and maintained manually — no built-in RLS management layer
- Bad, because more operational overhead — patching, monitoring, backup verification, security configuration are the team's responsibility
- Bad, because duplicates the database capability already bundled with Supabase Auth, adding unnecessary architectural complexity

### Option 4: MongoDB Atlas

**Description**: MongoDB Atlas document database with UK region hosting. Flexible schema for evolving data models with native JSON document storage.

- Good, because flexible schema accommodates evolving data models without migrations
- Good, because strong developer experience with native JSON document model and comprehensive SDKs
- Good, because UK region hosting available on Atlas
- Good, because built-in encryption at rest and in transit
- Neutral, because client-side field-level encryption available but more complex to implement than pgcrypto
- Bad, because less suited for the relational data model — ritual tracking has strong relational patterns (many-to-one with User Profile and Session Data) that are natural in SQL but require denormalisation or $lookup aggregation in MongoDB
- Bad, because no Row Level Security equivalent — per-user data isolation must be enforced entirely in application code, increasing the surface area for data isolation bugs
- Bad, because GDPR compliance documentation is less mature and less well-documented than PostgreSQL/Supabase ecosystem
- Bad, because does not integrate with Supabase Auth (ADR-005) — requires separate database client, connection management, and data synchronisation
- Bad, because more expensive than Supabase Pro for comparable capability (Atlas Dedicated clusters start at ~GBP 60/month)

## Data Model

The following entities are persisted in Supabase PostgreSQL with RLS policies ensuring per-user isolation:

### User Profile (DR-001)

| Attribute | Type | Required | Description | Constraints |
|-----------|------|----------|-------------|-------------|
| user_id | UUID | Yes | Unique identifier (from Supabase Auth) | Primary key |
| email_hash | String(64) | Yes | SHA-256 hashed email for identity | Not reversible |
| age_confirmed | Boolean | Yes | User confirmed 18+ | Must be true |
| consent_general | Timestamp | Yes | General consent granted | Not null |
| consent_research | Timestamp | No | Research consent (48hr delay) | Nullable |
| therapist_involved | Boolean | No | Working with therapist | Default null |

**RLS Policy**: `auth.uid() = user_id` — users can only access their own profile.

**Retention**: Active while account exists; deleted within 30 days of account deletion.

### Session Data (DR-002)

| Attribute | Type | Required | Description | Constraints |
|-----------|------|----------|-------------|-------------|
| session_id | UUID | Yes | Unique session identifier | Primary key |
| user_id | UUID | Yes | Foreign key to user profile | Indexed, FK |
| started_at | Timestamp | Yes | Session start time | Indexed |
| ended_at | Timestamp | Yes | Session end time | Not null after session ends |
| exchange_count | Integer | Yes | Number of user messages | Max 20 |
| phase_assessed | Enum | Yes | Limerence phase | initiation, crystallisation, deterioration, resolution |
| interventions_used | Array(String) | No | Intervention types applied | Field-level encrypted (pgcrypto) |
| action_commitment | Text | No | Post-session stated action | Max 500 chars, field-level encrypted |
| crisis_event | Boolean | No | Crisis detection triggered | Default false |

**RLS Policy**: `auth.uid() = user_id` — users can only access their own sessions.

**Retention**: Full session records deleted after 12 months; summary data (counts, phase, interventions) retained while account active.

### Ritual Tracking (DR-003)

| Attribute | Type | Required | Description | Constraints |
|-----------|------|----------|-------------|-------------|
| tracking_id | UUID | Yes | Unique record identifier | Primary key |
| user_id | UUID | Yes | Foreign key to user profile | Indexed, FK |
| ritual_name | String(100) | Yes | User-defined ritual label | Field-level encrypted (pgcrypto) |
| date | Date | Yes | Date of report | Indexed |
| slip_count | Integer | Yes | Times ritual performed | Min 0 |
| resist_count | Integer | No | Times urge resisted | Min 0, default 0 |
| session_id | UUID | Yes | Session in which reported | FK to Session Data |

**RLS Policy**: `auth.uid() = user_id` — users can only access their own ritual tracking data.

**Retention**: Summary data retained while account active; raw per-session data follows session retention policy (12 months).

### Self-Assessment Scores (DR-004)

| Attribute | Type | Required | Description | Constraints |
|-----------|------|----------|-------------|-------------|
| assessment_id | UUID | Yes | Unique record identifier | Primary key |
| user_id | UUID | Yes | Foreign key to user profile | Indexed, FK |
| session_id | UUID | Yes | Session of assessment | FK to Session Data |
| instrument | String(50) | Yes | Assessment instrument ID | e.g., "DLIM-5" |
| total_score | Integer | Yes | Aggregate score | Validated range per instrument |
| item_scores | JSON | No | Individual item responses | Structured per instrument schema |
| assessed_at | Timestamp | Yes | When completed | Indexed |

**RLS Policy**: `auth.uid() = user_id` — users can only access their own assessment scores.

**Retention**: Retained while account is active (longitudinal trend data is clinically valuable).
