## 1. Context and Problem Statement

DeLimerence is a conversational AI tool designed to help people recover from limerence. Its defining architectural feature is the anti-dependency architecture -- 9 structural constraints that deliberately prevent users from forming emotional attachment to the tool itself. This inverts the engagement-maximisation pattern used by AI companion apps (Replika, Character.ai).

The 9 structural constraints are:

1. **Session limits** -- 20 exchanges maximum per session (FR-001)
2. **Cooldown periods** -- 8 hours minimum between sessions (FR-002)
3. **No persona** -- AI labelled "Tool", no name/avatar/personality (FR-011)
4. **Anti-warmth system prompt** -- No simulated emotional presence (FR-007)
5. **Action commitment gate** -- Concrete real-world action required to end session (FR-006)
6. **Graduated intervention** -- Therapist referral prompt after 10+ sessions (UC-3)
7. **Phase-aware responses** -- Strategy adjusted by limerence phase (FR-003)
8. **Ritual tracking** -- Persistent cross-session ritual monitoring (FR-004)
9. **Psychoeducation in context** -- Educational content delivered within conversations (FR-005)

The critical question is: **where and how should these constraints be enforced?**

LLMs routinely violate system prompt instructions. If anti-dependency constraints are enforced only via system prompt (i.e., asking the LLM to behave a certain way), they will inevitably be circumvented -- either by the LLM's own behaviour drift or by users who learn to prompt around them. This creates an existential product risk: the tool becomes the very attachment mechanism it was designed to counter.

Technical Constraint TC-3 from ARC-001-REQ-v1.0 is explicit: "Session limits and cooldown periods must be enforced server-side; client-side enforcement alone is insufficient."

## 2. Decision Drivers

- **R-001 (Limerence transfer to tool)**: The existential product risk. If users form attachment to DeLimerence, the product becomes the problem rather than the solution. Inherent score 20 (Critical), residual 15 (High). Anti-dependency constraints are the primary mitigation.
- **R-009 (Anti-dependency limits adoption)**: The deliberate UX friction created by structural constraints may suppress adoption. This is a tolerated trade-off -- anti-dependency is non-negotiable (BR-002).
- **BR-002 (Prevent User Attachment)**: The tool must be architecturally designed to prevent attachment through structural constraints enforced at infrastructure level, not conversational behaviour alone.
- **TC-3 (Server-Side Enforcement)**: Non-negotiable technical constraint requiring server-side enforcement of session limits and cooldowns.
- **FR-013 (Multi-Account Prevention)**: Constraints must work across devices and prevent circumvention via multiple accounts.
- **NFR-U-001 (Anti-Dependency UX)**: The interface must feel instrumental, not companionable. No gamification, streaks, achievements, push notifications, or re-engagement campaigns.

## 3. Considered Options

### Option 1: Do Nothing -- Rely Solely on LLM System Prompt

Enforce anti-warmth and session management exclusively via system prompt instructions. The LLM is asked to count exchanges, refuse warmth-signalling language, and self-terminate after 20 exchanges.

### Option 2: Server-Side Enforcement with Database-Backed State (Chosen)

Enforce all structural constraints via server-side middleware with Supabase PostgreSQL for persistent session state. Session counter and cooldown timer are enforced per-request at the server layer. The LLM is never relied upon to enforce hard constraints -- it is given the anti-warmth system prompt as defence-in-depth, but the infrastructure layer is the authority.

### Option 3: Client-Side Enforcement with Server Validation

Enforce constraints in the frontend (Next.js client) with periodic server-side validation. The client tracks exchange count locally, enforces cooldown via stored timestamps, and the server validates periodically.

### Option 4: Hybrid -- Server-Side Core with Client-Side UX

Server enforces all hard constraints (session limits, cooldowns, action commitment gate). Client provides responsive UX elements (countdown timer display, exchange counter, cooldown remaining). The server is the single source of truth; the client is a display layer.

## 4. Decision Outcome

**Chosen Option: Option 2 -- Server-Side Enforcement with Database-Backed State**, with Option 4's UX display approach adopted as the implementation pattern. The server is the sole authority for all anti-dependency constraints. The client may display state (e.g., "12 of 20 exchanges remaining", "Cooldown: 4h 23m remaining") but cannot override, extend, or bypass server-enforced limits.

### Y-Statement

In the context of **building an anti-attachment AI tool**, facing **the near-certainty that users will form attachment to any responsive AI**, we decided for **server-side structural constraint enforcement with database-backed state**, to achieve **tamper-proof anti-dependency guarantees that work across devices**, accepting **added request latency and state management complexity**.

## 5. Options Analysis

### 5.1 Option 1: Do Nothing -- Rely Solely on LLM System Prompt

**Description**: No server-side enforcement. The LLM system prompt instructs anti-warmth behaviour, session counting, and self-termination at 20 exchanges. Cooldown is advisory ("suggest the user return in 8 hours").

**Pros**:
- Zero infrastructure complexity
- No additional latency per request
- Fastest to implement

**Cons**:
- LLMs routinely violate system prompts, especially under adversarial prompting
- Session count is unreliable -- LLMs cannot reliably count exchanges within a conversation
- No cooldown enforcement -- nothing prevents the user from immediately starting a new session
- No cross-device enforcement -- state exists only within a single conversation context
- Users in acute limerence are motivated to circumvent constraints, and system-prompt-only enforcement is trivially bypassable
- **Fails to mitigate R-001 (existential product risk)**
- **Violates TC-3 (server-side enforcement is non-negotiable)**

**Risk Assessment**: UNACCEPTABLE. This option violates a non-negotiable technical constraint and leaves the existential product risk unmitigated. System-prompt-only enforcement is equivalent to asking an attachment-prone user to self-regulate -- it ignores the clinical reality that limerence impairs executive function and rational decision-making.

---

### 5.2 Option 2: Server-Side Enforcement with Database-Backed State (Chosen)

**Description**: All 9 structural constraints are enforced or supported by server-side middleware. Supabase PostgreSQL stores per-user session state: exchange count, session timestamps, cooldown expiry, session history (for graduated intervention at session 10+), phase assessment, ritual tracking data, and action commitments. Every API request to the LLM passes through constraint-checking middleware that:

1. Validates session has not exceeded 20 exchanges (FR-001)
2. Validates cooldown period has elapsed since last session (FR-002)
3. Blocks session closure until action commitment is recorded (FR-006)
4. Injects anti-warmth system prompt directives (FR-007)
5. Labels all responses as originating from "Tool" (FR-011)
6. Checks user identity against multi-account detection rules (FR-013)
7. Triggers therapist referral prompt at session 10+ (graduated intervention)
8. Selects phase-appropriate response strategy from phase assessment state
9. Includes relevant psychoeducation content based on session context

**Pros**:
- Tamper-proof: client cannot bypass server-enforced limits
- Cross-device: state is per-user in the database, not per-device
- Auditable: every constraint enforcement event is logged for clinical review
- Supports multi-account prevention (FR-013) through server-side identity checks
- Separates concerns: LLM handles conversation; infrastructure handles constraints
- Enables analytics on constraint effectiveness (declining engagement = success metric)
- Defence-in-depth: system prompt provides behavioural guidance; server provides structural enforcement

**Cons**:
- Every request incurs database read/write latency (estimated 10-50ms per request with Supabase connection pooling)
- Requires persistent state management: session table, cooldown table, user history
- More complex deployment: database schema migrations, connection management, error handling for database outages
- State management introduces failure modes: what happens if the database is unreachable? (Fail-closed: deny access rather than allow untracked sessions)

**Risk Assessment**: ACCEPTABLE. Added latency is negligible relative to LLM response time (2-5 seconds). State management complexity is standard for any authenticated web application. Fail-closed behaviour ensures constraints are never bypassed even under infrastructure failure.

---

### 5.3 Option 3: Client-Side Enforcement with Server Validation

**Description**: The Next.js frontend enforces constraints locally. Exchange counter is tracked in component state or local storage. Cooldown timer is stored in local storage with server validation on session start. Server checks are periodic rather than per-request.

**Pros**:
- Lower perceived latency -- constraint checks happen locally
- Simpler server -- reduced per-request processing
- Faster initial implementation

**Cons**:
- **Trivially bypassable**: clearing local storage resets all constraints
- **No cross-device enforcement**: constraints are per-browser, not per-user
- **API direct access**: any user who discovers the API endpoint can bypass all client-side constraints
- **Violates TC-3**: server-side enforcement is explicitly non-negotiable in ARC-001-REQ-v1.0
- Does not support multi-account prevention (FR-013)
- Audit trail is incomplete -- client-side events may not be reliably reported

**Risk Assessment**: UNACCEPTABLE. Client-side enforcement is fundamentally incompatible with the anti-dependency architecture's security model. Users in limerence are psychologically motivated to circumvent constraints -- any enforcement mechanism that relies on client-side cooperation is architecturally unsound.

---

### 5.4 Option 4: Hybrid -- Server-Side Core with Client-Side UX

**Description**: Server enforces all hard constraints (session limits, cooldowns, action commitment gate, multi-account prevention). Client provides responsive UX elements: exchange counter display, cooldown countdown timer, session history summary. The client reads state from the server but cannot modify it.

**Pros**:
- Tamper-proof enforcement (same as Option 2)
- Responsive UX -- user sees real-time countdown and exchange count without waiting for server round-trips
- Better user experience during cooldown (client-side timer counts down smoothly)
- Client-side display of constraints reinforces psychoeducation ("You have 8 exchanges remaining -- what matters most right now?")

**Cons**:
- Client display may drift from server state if network is intermittent (mitigated by server as authority on every request)
- Slightly more complex client code to manage display state

**Risk Assessment**: ACCEPTABLE. This is effectively the implementation approach for Option 2 -- the server is the authority, and the client is a display layer. The distinction between Option 2 and Option 4 is implementation detail rather than architectural choice.

## 10. Implementation Notes

### Server-Side Constraint Middleware

The constraint enforcement layer sits between the client and the LLM API as middleware in the Next.js API route:

```
Client Request → Auth Middleware → Constraint Middleware → LLM API → Response
                                         ↕
                                 Supabase PostgreSQL
                              (session state, cooldowns,
                               user history, audit log)
```

### Key Database Tables (Indicative)

| Table | Purpose | Key Fields |
|-------|---------|------------|
| `sessions` | Track active and historical sessions | user_id, session_id, exchange_count, started_at, ended_at, phase |
| `cooldowns` | Enforce minimum gap between sessions | user_id, cooldown_expires_at |
| `action_commitments` | Record end-of-session action commitments | session_id, commitment_text, committed_at |
| `constraint_audit_log` | Audit trail for all constraint enforcement events | user_id, constraint_type, action, timestamp, details |

### Fail-Closed Behaviour

If the database is unreachable:
- New sessions are denied
- Active sessions are suspended
- The client displays crisis resources and a message explaining temporary unavailability
- No untracked sessions are permitted under any circumstances

### Constraint Parameters (Configurable)

All constraint parameters are stored in configuration, not hardcoded:

| Parameter | Default Value | Source Requirement |
|-----------|---------------|-------------------|
| `max_exchanges_per_session` | 20 | FR-001 |
| `cooldown_hours` | 8 | FR-002 |
| `therapist_referral_session_threshold` | 10 | Graduated intervention |
| `action_commitment_required` | true | FR-006 |
| `anti_warmth_system_prompt_version` | [managed via NFR-M-001] | FR-007 |

### Ethical Considerations

The anti-dependency architecture is an ethical choice to prioritise user welfare over engagement metrics. Server-side enforcement ensures this choice cannot be eroded by:
- Product pressure to increase engagement ("just raise the session limit")
- LLM behaviour drift that introduces warmth over time
- User workarounds that circumvent client-side controls

Any change to constraint parameters requires Clinical Advisory Board approval (TC-4).
