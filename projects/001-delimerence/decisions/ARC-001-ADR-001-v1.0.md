## Context and Problem Statement

DeLimerence is a conversational AI tool designed to help people recover from limerence -- obsessive romantic attachment characterised by intrusive thoughts, compulsive rituals, and disrupted executive function. It is the deliberate inversion of AI companion apps: where Replika and Character.ai deepen emotional attachment through intermittent reinforcement and simulated emotional presence, DeLimerence systematically loosens attachment using CBT, ERP, and psychoeducation grounded in clinical literature [LM35-C1].

The entire conversational capability of the tool depends on selecting a foundation LLM that can conduct nuanced, phase-aware, psychoeducation-rich conversations within strict anti-dependency constraints. This is the most critical technology decision for the project.

The selected LLM must:

1. **Adhere to anti-warmth system prompts** -- the model must reliably follow instructions to avoid empathetic mirroring, emotional validation loops, and warmth patterns that could trigger attachment [FR-007]
2. **Deliver clinically accurate psychoeducation** -- conversations must incorporate CBT/ERP techniques adapted from the Wyant (2021) case study without making diagnostic claims [FR-005, FR-008]
3. **Operate within a zero-retention DPA** -- no conversation data may be stored at rest by the LLM provider, as sessions contain special category health data under UK GDPR Article 9 [INT-001]
4. **Provide UK data residency** -- all data processing must occur within UK borders to satisfy ICO requirements and the project's data residency constraint [TC-1]
5. **Respond within 5 seconds (p95)** -- latency must support natural conversational flow without encouraging compulsive refreshing behaviour [NFR-P-001]
6. **Degrade gracefully on provider outage** -- the system must fall back to static crisis resources rather than presenting a broken interface [NFR-A-003]

The decision must also account for risks R-004 (LLM produces clinically harmful response) and R-010 (LLM provider DPA inadequate for special category health data), both identified in ARC-001-RISK-v1.0.

## Considered Options

### Option 1: Do Nothing -- No LLM, Static Content Only

**Description**: Do not integrate an LLM. Deliver DeLimerence as a static content tool with pre-written psychoeducation articles, ritual tracking forms, and self-assessment questionnaires -- no conversational capability.

**Cost Breakdown**:

| Cost Item | Year 1 | Year 2 | Year 3 | 3-Year TCO |
|-----------|--------|--------|--------|------------|
| LLM API costs | GBP 0 | GBP 0 | GBP 0 | GBP 0 |
| Integration development | GBP 0 | GBP 0 | GBP 0 | GBP 0 |
| **Total** | **GBP 0** | **GBP 0** | **GBP 0** | **GBP 0** |

**Pros**:

- Zero API cost and zero vendor dependency
- No data processing risk from LLM provider
- Simpler architecture, faster to build
- No risk of clinically harmful generated responses (R-004 eliminated)

**Cons**:

- No conversational capability -- the core value proposition of DeLimerence is conversational intervention at the point of crisis (3am, acute distress)
- Static content cannot adapt to user's limerence phase, specific rituals, or individual circumstances
- Cannot deliver CBT/ERP techniques in a responsive, personalised format
- Tool becomes a glorified self-help website -- does not meet BR-001
- Fails to address FR-003 (phase assessment), FR-005 (psychoeducation delivery), FR-007 (anti-warmth behaviour), FR-008 (ABCDE restructuring), FR-009 (LLM integration)
- Renders the anti-dependency architecture largely irrelevant (no conversation to constrain)

**Assessment**: **REJECTED** -- Eliminates the core capability that justifies the project's existence. DeLimerence without conversational AI is not DeLimerence.

### Option 2: Anthropic Claude via AWS Bedrock (CHOSEN)

**Description**: Use Anthropic Claude Sonnet 4.6 as the foundation LLM, accessed via AWS Bedrock eu-west-2 (London) regional endpoint. This provides UK data residency, zero-retention data processing, and access to Claude's Constitutional AI safety training through a managed AWS service.

**Vendor**: Anthropic (San Francisco, founded 2021) via AWS Bedrock  
**Model**: Claude Sonnet 4.6 (production), with Haiku 4.5 fallback for non-clinical interactions  
**Region**: AWS Bedrock eu-west-2 (London)

**Cost Breakdown**:

|-----------|--------|--------|--------|------------|
| API usage (Claude Sonnet 4.6) | GBP 9,600 | GBP 19,200 | GBP 38,400 | GBP 67,200 |
| Bedrock regional premium (10%) | GBP 960 | GBP 1,920 | GBP 3,840 | GBP 6,720 |
| Integration development | GBP 4,000 | GBP 0 | GBP 0 | GBP 4,000 |
| **Total** | **GBP 14,560** | **GBP 21,120** | **GBP 42,240** | **GBP 77,920** |

**Usage Assumptions** (from ARC-001-RSCH-v1.0):

- Year 1: 1,000 daily sessions x 20 exchanges x ~2,000 tokens/exchange = 40M tokens/day
- Year 3: 20,000 daily sessions at scale
- Claude Sonnet 4.6 pricing: $3/M input, $15/M output (approx 60:40 input:output ratio)
- Prompt caching reduces system prompt costs by up to 90%

**Pros**:

- **Best safety training**: Constitutional AI approach produces the most safety-aware responses of any frontier model -- critical for R-004 mitigation
- **Strong system prompt adherence**: Claude reliably follows anti-warmth constraints in system prompts, reducing risk of empathetic warmth leakage that could trigger attachment [FR-007]
- **UK data residency**: AWS Bedrock eu-west-2 provides inference on UK soil, satisfying ICO data residency expectations [TC-1]
- **Zero-retention DPA**: Bedrock zero-retention option means no conversation data is stored at rest by the provider [INT-001]
- **Compliance certifications**: ISO 27001:2022, ISO 42001 (AI Management System), SOC 2 Type I & II
- **Anthropic DPA**: Updated January 2026, automatically covers UK GDPR [R-010]
- **Prompt caching**: Up to 90% cost reduction on system prompt tokens (system prompt is large due to anti-warmth rules and phase-aware instructions)
- **1M token context window**: Sufficient for full session context plus persistent data summary
- **Streaming support**: Real-time conversational UX without perceived latency
- **Tool use / function calling**: Structured data extraction for ritual tracking without fragile parsing
- **Batch API**: 50% discount for async workloads (outcome report generation)
- **Strong vendor maturity**: Series E funding, $61.5B valuation (2025), substantial runway

**Cons**:

- **Dual vendor dependency**: Dependent on both Anthropic (model provider) and AWS (infrastructure provider) -- failure or policy change at either affects the service
- **10% Bedrock regional premium**: UK regional endpoint costs 10% more than US regions
- **Model update risk**: Anthropic model updates may change conversational behaviour, requiring clinical re-validation of system prompt and safety layer per NFR-M-001
- **Token-based pricing**: Cost prediction is harder than subscription model; usage spikes could cause budget overruns
- **No direct Anthropic UK endpoint**: Must use Bedrock rather than Anthropic's direct API for UK data residency

**Assessment**: **CHOSEN** -- Best alignment with safety requirements, UK data residency, and anti-warmth system prompt adherence. The dual vendor dependency is an accepted trade-off for the combination of Constitutional AI safety and UK-soil inference.

**Pros**:

- Strong model capability and largest ecosystem
- Azure UK South provides UK data residency
- Zero data retention available for API usage
- UK GDPR-compliant DPA
- SOC 2 Type II certified
- Competitive pricing (GPT-4o at $2.50/M input tokens) -- GBP 23,520 cheaper over 3 years than Claude via Bedrock

**Cons**:

- **Less safety-focused training**: GPT-4 lacks Claude's Constitutional AI approach; safety training is effective but not as deeply embedded in the model architecture
- **Weaker system prompt adherence for anti-warmth**: Historically weaker at maintaining strict anti-warmth constraints over extended conversations -- risk of warmth leakage that could trigger attachment in vulnerable users [FR-007, R-004]
- **Azure OpenAI requires separate application process**: Access is not self-service; approval process adds timeline risk
- **Data processing terms less clear for health data**: While DPA covers GDPR, specific provisions for special category health data (Article 9) are less explicit than Anthropic's
- **OpenAI governance instability**: Significant corporate governance changes create uncertainty about long-term direction and safety commitment
- **Model update unpredictability**: Safety characteristics may shift with model updates; less transparency about changes than Anthropic

**Assessment**: **REJECTED** -- While cheaper and capable, the weaker system prompt adherence for anti-warmth constraints is a disqualifying factor for a tool designed to prevent attachment in psychologically vulnerable users. The GBP 23,520 cost saving over 3 years does not justify the increased clinical safety risk.

**Description**: Fine-tune an open-source foundation model (e.g., Llama, Mistral) on clinical limerence data to create a purpose-built LLM for anti-dependency conversations. Deploy on owned GPU infrastructure.

**Technology Stack**: PyTorch, Hugging Face Transformers, RLHF pipeline, GPU infrastructure (A100 cluster)

**Cost Breakdown**:

**Pros**:

- Full control over model behaviour, anti-warmth tuning, and clinical accuracy
- No vendor dependency for core conversational capability
- IP ownership of fine-tuned model
- UK data residency guaranteed (own infrastructure)

**Cons**:

- **Prohibitively expensive**: GBP 564,000 3-year TCO is 7.2x the cost of Claude via Bedrock
- **18+ month timeline**: Conflicts directly with Q4 2026 beta target (BR-001, BC-1) -- cannot deliver on time
- **Requires ML team that does not exist**: The project currently has no ML engineering capability; hiring and onboarding would add further delay
- **Clinical data scarcity**: Fine-tuning requires substantial training data; the clinical evidence base for limerence treatment consists of a single published case study (Wyant, 2021) [LM35-C4]
- **Safety alignment is unsolved**: Aligning a custom model for mental health safety is an active research problem; no guarantee of achieving Claude-level safety
- **Maintenance burden**: Ongoing retraining and safety updates require permanent ML engineering capacity

**Assessment**: **REJECTED** -- Timeline, cost, and clinical data constraints make this infeasible for the current project phase. May be reconsidered if DeLimerence reaches scale (100,000+ daily sessions) and the clinical evidence base matures sufficiently for fine-tuning.

## Decision Outcome

### Chosen Option

**Option 2: Anthropic Claude via AWS Bedrock**

### Y-Statement

In the context of **building a conversational AI tool for limerence recovery**, facing the need for **nuanced anti-warmth conversations with UK data residency**, we decided for **Anthropic Claude via AWS Bedrock**, to achieve **safety-aware clinical conversations with zero-retention data processing**, accepting **vendor dependency on both Anthropic and AWS**.

### Rationale

The decision prioritises clinical safety over cost optimisation. Claude's Constitutional AI training and demonstrated system prompt adherence for anti-warmth constraints directly address the most critical product risk: a tool designed to loosen attachment must not exhibit conversational warmth that could trigger new attachment in psychologically vulnerable users (R-004, R-001).

The GBP 23,520 premium over OpenAI GPT-4 via Azure (3-year TCO) is justified by:

1. **Stronger safety baseline** -- Constitutional AI provides more reliable anti-warmth behaviour, reducing the burden on the external safety layer (ADR-003)
2. **Better system prompt adherence** -- critical for maintaining anti-warmth constraints across 20-exchange sessions
3. **Clearer data processing terms** -- Anthropic's DPA explicitly covers GDPR with zero-retention, reducing R-010
4. **Simpler access model** -- Bedrock is self-service; Azure OpenAI requires separate application approval

The custom build option (Option 4) is rejected on timeline (18+ months vs Q4 2026 target), cost (7.2x premium), and feasibility (no ML team, insufficient training data) grounds.

## Consequences

### Positive

- **Safety-first LLM**: Claude's Constitutional AI training provides the strongest baseline for anti-warmth, clinically safe conversations, directly supporting the anti-dependency architecture
- **UK data residency achieved**: AWS Bedrock eu-west-2 satisfies ICO requirements and project data residency constraint (TC-1)
- **Zero-retention DPA**: No conversation data stored at rest by the LLM provider, protecting special category health data under Article 9
- **Compliance certifications**: ISO 27001, ISO 42001, SOC 2 provide evidence for DPIA and regulatory submissions
- **Prompt caching reduces cost**: 90% savings on system prompt tokens (significant given the large anti-warmth system prompt)
- **Streaming enables natural UX**: Real-time token streaming prevents perceived latency from triggering compulsive refresh behaviour
- **Rapid time to market**: 2 person-weeks integration development supports Q4 2026 beta target

### Negative

- **Dual vendor dependency**: Service depends on both Anthropic (model) and AWS (infrastructure); failure or policy change at either affects DeLimerence
- **Model update re-validation cost**: Each Anthropic model update requires clinical re-validation of system prompt behaviour -- estimated 2-3 person-days per update
- **10% regional premium**: UK Bedrock endpoint costs 10% more than US regions, adding GBP 6,720 over 3 years
- **Token-based cost uncertainty**: Usage-based pricing makes budget forecasting harder than fixed subscription models; unexpected usage spikes could exceed budget

### Neutral

- **Abstraction layer required**: Integration should use an abstraction layer (adapter pattern) to enable future provider switching -- this is good engineering practice regardless of this decision
- **Haiku fallback option**: Non-clinical interactions (ritual data entry, settings) can use cheaper Claude Haiku 4.5 model, but this adds routing complexity
- **Future re-evaluation**: If DeLimerence reaches 100,000+ daily sessions, the custom LLM option (Option 4) should be re-evaluated as API costs may exceed infrastructure costs at that scale
