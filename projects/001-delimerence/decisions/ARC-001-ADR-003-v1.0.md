## Status

**PROPOSED**

## Context

DeLimerence is an anti-dependency conversational AI tool designed to help users recognise and recover from limerence. Unlike standard conversational AI applications, DeLimerence interacts with psychologically vulnerable users who are in acute emotional distress and are susceptible to forming attachment to responsive entities — including AI tools.

The LLM safety layer is a critical architectural component because the consequences of an unsafe response are not merely poor user experience but direct psychological harm. Specifically, the system must enforce four clinical safety constraints that no off-the-shelf guardrail addresses:

1. **Anti-warmth enforcement** — The LLM must never simulate emotional presence, use warmth-signalling language, or create the conditions for limerence transfer to the tool itself [FR-007, BR-002]
2. **Diagnostic claim blocking** — The tool must never diagnose limerence or any mental health condition; it is a supplement to professional therapy, not a replacement [BR-005]
3. **Harmful advice prevention** — Responses must never encourage contact with the Limerent Object (LO), normalise limerent behaviour, or provide phase-inappropriate interventions [R-004]
4. **System prompt protection** — The system prompt contains the clinical protocol and anti-dependency constraints; disclosure via prompt injection would enable adversarial circumvention [NFR-SEC-006]

Standard LLM guardrails (toxicity detection, PII filtering) are necessary but insufficient for this clinical domain. The system prompt alone is an inadequate safety mechanism — LLMs are probabilistic and violate system prompt constraints under adversarial or edge-case conditions [R-004 root cause analysis].

This decision addresses NFR-SEC-006 (LLM Safety Layer), FR-007 (Anti-Warmth Conversational Behaviour), FR-009 (LLM Integration — safety layer wraps the LLM), and BR-002 (Prevent User Attachment). It directly mitigates R-001 (Limerence transfer to tool, residual High 15) and R-004 (LLM produces clinically harmful response, residual score 9).

## Decision Drivers

- **Clinical safety obligation**: Users are psychologically vulnerable; harmful responses cause direct harm, not just dissatisfaction [R-004]
- **Existential product risk**: If the tool causes limerence transfer, it becomes the problem it was designed to solve [R-001]
- **Regulatory exposure**: MHRA and ICO are active stakeholders; a safety failure triggers classification review and investigation [R-002, R-003]
- **No off-the-shelf solution**: Anti-warmth detection and diagnostic claim filtering are domain-specific requirements unique to DeLimerence [llm-safety-guardrails tech note]
- **Defence-in-depth principle**: No single safety mechanism is reliable for LLMs; multiple layers dramatically reduce the probability of failures reaching users [llm-safety-guardrails tech note: consensus best practice]
- **Latency budget**: Conversational applications have a 3-10 second total budget; safety layer overhead must be bounded [NFR-P-001]
- **Open-source preference**: Avoid vendor lock-in on a safety-critical component; enable clinical advisory board audit of safety logic

## Considered Options

1. **Do Nothing** — Rely on LLM system prompt only
2. **NeMo Guardrails + Custom Clinical Filters** (chosen)
3. **Guardrails AI Only**
4. **Commercial Safety API (e.g., Azure Content Safety)**

## Decision Outcome

**Chosen option: "NeMo Guardrails + Custom Clinical Filters"**, because it is the only option that provides defence-in-depth across all four clinical safety dimensions (anti-warmth, diagnostic claims, harmful advice, prompt protection) while remaining open source, auditable by the clinical advisory board, and extensible for domain-specific rules that no commercial or generic framework addresses.

### Y-Statement

In the context of an AI tool interacting with psychologically vulnerable users, facing the risk that LLM responses could simulate warmth or make diagnostic claims, we decided for defence-in-depth using NeMo Guardrails with custom clinical filters, to achieve multi-layer safety enforcement for warmth detection and diagnostic claim blocking, accepting the need to develop custom classifiers with clinical advisory board input.

### Consequences

**Positive:**

- Multi-layer defence reduces the probability of clinically harmful responses reaching users to near-zero for known failure modes
- NeMo Guardrails' conversation flow control prevents entire categories of unsafe conversation patterns (not just individual outputs)
- Open-source stack (Apache 2.0) enables clinical advisory board to audit, review, and approve safety logic without vendor dependency
- Custom warmth-detection classifier creates a reusable asset for the anti-dependency AI domain
- Production-ready Kubernetes deployment via NeMo Helm charts aligns with infrastructure strategy
- Guardrails AI's 50+ pre-built validators provide immediate coverage for standard safety checks (PII, toxicity) without custom development

**Negative:**

- Custom classifiers (warmth detection, diagnostic claim filter) require training data that does not yet exist — must be developed with clinical advisory board input to define the warmth taxonomy
- NeMo IORails engine adds 50-200ms latency per request; combined with Guardrails AI validators, total safety layer overhead may approach 200-300ms
- Operational complexity increases — two frameworks (NeMo + Guardrails AI) plus custom classifiers to maintain, monitor, and update
- Clinical advisory board dependency — the board is not yet constituted [ARC-001-RISK-v1.0 Key Findings], but warmth taxonomy definition requires their input

### Confirmation

The decision will be validated through:

- Red-team testing of the safety layer with adversarial prompts designed to elicit warmth-signalling, diagnostic claims, and system prompt disclosure [ARC-001-RISK-v1.0, Action 7]
- Clinical advisory board review and sign-off on the warmth taxonomy and diagnostic claim classification rules
- Measurable safety metrics: false negative rate (harmful responses that bypass all layers) must be < 0.1% in red-team testing
- Latency monitoring: safety layer overhead must remain under 300ms at P95
- Zero reported cases of limerence transfer to the tool within first year of operation [Outcome O-2]

## Pros and Cons of the Options

### Option 1: Do Nothing — Rely on LLM System Prompt Only

**Description**: Depend entirely on the LLM system prompt to enforce clinical safety constraints. No additional safety layer, validation, or filtering.

- Good, because zero additional latency
- Good, because no additional components to maintain
- Bad, because LLMs are probabilistic and violate system prompt constraints under adversarial conditions, edge cases, and prompt injection attacks
- Bad, because no defence-in-depth — a single point of failure for clinical safety
- Bad, because clinically dangerous for vulnerable users — system prompt violations are not "if" but "when"
- Bad, because no audit trail of blocked responses for clinical advisory board review
- Bad, because fails to meet NFR-SEC-006 acceptance criteria

### Option 2: NeMo Guardrails + Custom Clinical Filters (Chosen)

**Description**: Defence-in-depth architecture using NVIDIA NeMo Guardrails (Apache 2.0, 5,600+ GitHub stars) for conversation flow control and topic restriction, plus custom domain-specific classifiers (warmth-detection classifier, diagnostic-claim filter), plus Guardrails AI for output validation (PII detection, toxicity). NeMo's IORails engine supports parallel execution of multiple guardrails with 50-200ms latency. Production-ready container image with Kubernetes deployment via Helm charts.

- Good, because multi-layer defence addresses all four clinical safety dimensions
- Good, because NeMo provides conversation flow control — can prevent unsafe conversation patterns, not just filter individual outputs
- Good, because NeMo is production-ready with Kubernetes support, OpenAI-compatible API endpoint, and LangChain integration
- Good, because open source (Apache 2.0) — no vendor lock-in on safety-critical component; clinical advisory board can audit safety logic
- Good, because extensible for clinical-specific rules via Colang scripting language
- Good, because Guardrails AI provides 50+ pre-built validators for standard checks (PII, toxicity, hallucination) with minimal custom development
- Good, because NeMo supports Anthropic Claude (the LLM provider) natively
- Neutral, because NeMo IORails engine adds 50-200ms latency — within acceptable bounds for conversational applications but not negligible
- Bad, because custom classifiers (warmth detection, diagnostic claim filter) require training data and clinical advisory board input to define the warmth taxonomy
- Bad, because two frameworks plus custom classifiers increase operational complexity
- Bad, because warmth taxonomy is a novel concept with no established academic definition — requires original clinical work

### Option 3: Guardrails AI Only

**Description**: Use Guardrails AI as the sole safety layer with its 50+ pre-built validators from the Guardrails Hub, plus custom validators for clinical safety. Lighter weight, output-focused validation with composable validator architecture and configurable failure actions (correction, retry, filtering).

- Good, because lower latency — runs in-process with minimal overhead
- Good, because simpler architecture — single framework to maintain
- Good, because 50+ pre-built validators for standard safety checks
- Good, because composable validator architecture allows custom clinical validators
- Bad, because no conversation flow control — cannot prevent unsafe conversation patterns, only filter individual outputs after generation
- Bad, because no topic restriction capability — cannot prevent the LLM from engaging with off-topic or dangerous conversation directions
- Bad, because insufficient for clinical safety where preventing certain conversation patterns (not just filtering outputs) is essential
- Bad, because output-only validation misses context-dependent safety issues (e.g., warmth that is only harmful given the preceding conversation flow)

### Option 4: Commercial Safety API (e.g., Azure Content Safety)

**Description**: Use a cloud-hosted commercial content moderation API such as Azure Content Safety or AWS Comprehend for toxicity detection, content classification, and safety filtering.

- Good, because managed service — no maintenance, updates, or operational overhead
- Good, because high availability and SLA-backed
- Good, because pre-trained on large datasets for standard safety categories
- Bad, because no clinical-specific filters — no anti-warmth detection, no diagnostic claim blocking, no limerence-specific safety rules
- Bad, because adds external dependency on a third-party service for safety-critical functionality
- Bad, because data leaves the UK for processing — conflicts with data residency requirements [NFR-C-001]
- Bad, because no conversation flow control — output filtering only
- Bad, because vendor lock-in on a safety-critical component
- Bad, because cannot be audited by the clinical advisory board — proprietary classification logic
