Use Supabase Auth for User Authentication and Identity Management

### Escalation Level

## Context and Problem Statement

DeLimerence is a conversational AI tool that helps people recognise and recover from limerence using CBT, ERP, and psychoeducation. The tool requires user authentication to enable session persistence, per-user cooldown enforcement, ritual tracking across sessions, and consent management. These capabilities are foundational -- without authenticated identity, the anti-dependency architecture cannot enforce its structural constraints (e.g., the 2-hour absolute session timeout from NFR-SEC-001 or the mandatory cooldown periods from FR-002).

The user population presents unique authentication design constraints. Users are psychologically vulnerable with potentially compromised executive function [ARC-001-REQ-v1.0, FR-010]. Consent flows must be designed for this population: plain language, no dark patterns, clear opt-out at every stage. The system processes special category data under UK GDPR Article 9 (mental health disclosures, limerent rituals, relationship information), making the identity layer the first line of defence against data breach [R-003].

Multi-account prevention is required to stop users from circumventing cooldown periods by creating duplicate accounts [FR-013], which would undermine the anti-dependency architecture. The system is adult-only (18+) at launch, requiring age confirmation during account creation.

The authentication solution must support role-based access control with strict data isolation between users, clinical reviewers, admins, and researchers [NFR-SEC-002].

## Decision Drivers

1. **Vulnerable user population** -- Authentication must be simple, accessible, and designed for users with compromised executive function. Complex identity federation or multi-step flows are contraindicated.
2. **Data isolation** -- Special category health data requires strict per-user isolation. Row Level Security at the database layer is preferable to application-layer enforcement alone.
3. **Cooldown enforcement** -- Per-user cooldowns are a structural constraint of the anti-dependency architecture. Identity must persist across sessions and resist circumvention.
4. **Consent architecture** -- UK GDPR Article 9 requires explicit consent for special category processing. Consent must be bound to authenticated identity.
5. **Startup economics** -- The project is founder-led with a 3-year TCO target of GBP 275,000 [ARC-001-RSCH-v1.0]. Authentication cost must be proportionate.
6. **Time to market** -- Public beta target is Q4 2026 [Goal G-1]. The authentication solution must minimise integration effort.

## Considered Options

### Option 1: Do Nothing (Anonymous Access)

Allow users to access the tool without authentication. Session state stored in browser local storage only.

**Pros:**
- Zero implementation effort for authentication
- Lowest barrier to entry for distressed users
- No personal data collection required

**Cons:**
- Cannot enforce per-user cooldown periods -- undermines anti-dependency architecture
- No session persistence across devices or browser sessions
- No ritual tracking over time -- cannot demonstrate clinical outcomes (Goal G-2, Outcome O-1)
- No consent management -- violates BR-004 and UK GDPR Article 9 requirements
- No data isolation -- cannot implement RBAC (NFR-SEC-002)
- No multi-account prevention (FR-013 becomes impossible)
- Violates BR-002 (prevent user attachment via structural constraints that require identity)
- Violates FR-002 (mandatory cooldown periods require user identification)

**Assessment:** Not viable. Anonymous access is fundamentally incompatible with the anti-dependency architecture and regulatory requirements.

---

### Option 2: Supabase Auth (CHOSEN)

Use Supabase Auth as the authentication provider, bundled with Supabase PostgreSQL as the primary database. Supabase Auth provides email/password authentication, optional MFA (TOTP), custom claims for RBAC, and integrates natively with PostgreSQL Row Level Security.

**Pricing:** GBP 25/month (Pro tier) includes both authentication and database. 50,000 free MAUs on Free tier; 100,000 MAUs on Pro tier.

**Pros:**
- Bundled authentication + PostgreSQL database reduces integration complexity and procurement overhead
- 50,000 free MAUs covers projected Year 1 user base
- Row Level Security (RLS) provides database-layer data isolation -- defence in depth beyond application-layer checks
- Email/password + optional TOTP MFA meets NFR-SEC-001 requirements
- Custom claims support RBAC for User, Clinical Reviewer, Admin, and Researcher roles (NFR-SEC-002)
- GDPR compliant with Data Processing Agreement available
- SOC 2 Type 2 certified (Team tier and above)
- Open-source core (MIT/Apache 2.0) provides genuine exit strategy via self-hosting
- Comprehensive SDKs (JavaScript, Python) with strong developer experience
- Session management APIs support custom timeout logic (30-minute inactivity, 2-hour absolute)
- Startup-friendly pricing aligns with founder-led cost constraints

**Cons:**
- Vendor dependency on Supabase for a critical path capability
- No dedicated UK data centre -- closest is EU (Frankfurt); may not satisfy strict UK-soil requirements for health data
- Relatively young company (founded 2020) -- vendor viability consideration
- Less enterprise-grade than Auth0/Okta for complex identity scenarios
- Limited identity provider options compared to Auth0 (fewer social login providers)
- No built-in multi-account prevention -- device fingerprinting requires custom application logic
- Session timeout customisation requires application-layer middleware, not natively configurable
- SOC 2 Type 2 only available on Team tier (GBP 599/month), not Pro tier

---

### Option 3: Auth0

Use Auth0 (Okta) as a dedicated enterprise identity platform, with Supabase or another PostgreSQL provider as a separate database.

**Pricing:** Free for up to 7,500 MAU. Professional plan starts at approximately GBP 200/month for 1,000+ MAU. Enterprise pricing is custom.

**Pros:**
- Most mature identity platform on the market
- Extensive identity provider support (social, enterprise SSO, passwordless)
- Comprehensive compliance certifications (SOC 2 Type 2, ISO 27001, HIPAA BAA)
- Built-in Adaptive MFA with risk-based authentication
- Extensive customisation of login flows (Universal Login)
- Large ecosystem of integrations and extensions
- Dedicated UK/EU data residency options

**Cons:**
- Expensive at scale -- GBP 200+/month for 1,000+ MAU, escalating rapidly
- Separate from database -- requires additional integration layer and separate procurement
- Over-engineered for a startup with a single user type and simple RBAC model
- Complex configuration for what is fundamentally an email/password + optional MFA use case
- No database-layer data isolation -- RLS must be implemented separately
- Enterprise features (Adaptive MFA, SSO) are not needed for DeLimerence's user population
- Adds a second vendor dependency without proportionate benefit

---

### Option 4: Custom Auth (NextAuth.js)

Build authentication using NextAuth.js (Auth.js) with Supabase as the database backend only. All authentication logic, session management, and MFA handled in application code.

**Pros:**
- Full control over authentication flow and session management
- No authentication vendor dependency -- only database dependency
- NextAuth.js is open source (ISC license) with strong community
- Can implement session timeouts (30-minute, 2-hour) natively without middleware
- Can design consent flow exactly to specification without provider constraints
- Lower ongoing cost -- no per-MAU pricing for authentication

**Cons:**
- Significantly more development effort for authentication, session management, and security hardening
- MFA must be implemented separately (TOTP libraries, backup codes, recovery flows)
- Security responsibility falls entirely on the development team -- no security audit trail from a specialised provider
- Must implement CSRF protection, rate limiting, brute force protection, and credential stuffing defence
- No SOC 2 or equivalent certification for the authentication layer
- Session management edge cases (token refresh, concurrent sessions, device management) add complexity
- Delays time to market relative to managed solutions
- Increases the attack surface for R-003 (UK GDPR special category data breach)

## Decision Outcome

### Chosen Option

**Option 2: Supabase Auth**, because it provides the best balance of cost, integration simplicity, and data isolation for a founder-led startup processing special category health data.

### Y-Statement

In the context of **authenticating vulnerable users for session-persistent limerence recovery**, facing the need for **per-user cooldown enforcement and consent management**, we decided for **Supabase Auth bundled with PostgreSQL**, to achieve **low-cost identity management with built-in data isolation via Row Level Security**, accepting **vendor dependency on Supabase**.

### Consequences

**Positive:**
- Single vendor for authentication and database reduces integration complexity and operational overhead
- Row Level Security provides defence-in-depth data isolation at the database layer, directly mitigating R-003
- GBP 25/month Pro tier is proportionate to a founder-led project, well within the GBP 275,000 3-year TCO envelope
- Open-source core provides a credible exit path if Supabase viability becomes a concern
- 50,000+ free MAUs means authentication cost is effectively zero during Year 1

**Negative:**
- Vendor dependency on Supabase for both authentication and database -- a single point of failure
- No UK data residency -- EU (Frankfurt) is the closest option; this must be assessed in the DPIA (Action 5 from ARC-001-RISK-v1.0)
- Multi-account prevention (FR-013) requires custom application logic -- not provided by Supabase
- Session timeout enforcement (30-minute inactivity, 2-hour absolute) requires application-layer middleware
- SOC 2 Type 2 certification requires Team tier (GBP 599/month), which may be needed before public beta

**Neutral:**
- MFA is optional at launch, aligning with NFR-SEC-001 -- but should be encouraged for users processing sensitive data
- RBAC via custom claims is adequate for four roles but would need re-evaluation if role complexity increases

## More Information

### Implementation Notes

1. **Row Level Security Policies**: RLS policies must be defined for all tables containing user data. The default policy must deny all access; explicit policies grant access based on `auth.uid()` matching the row owner. Clinical Reviewer and Researcher roles access only anonymised aggregate views, never individual session data.

2. **Session Timeout Middleware**: Supabase Auth does not natively enforce custom session timeouts. Application middleware must:
   - Track last activity timestamp per session
   - Enforce 30-minute inactivity timeout (revoke session token)
   - Enforce 2-hour absolute session timeout (aligns with anti-dependency architecture)
   - Require re-authentication for sensitive operations (account deletion, consent changes, data export)

3. **Multi-Account Prevention**: Supabase enforces email uniqueness at the database level. Additional measures required:
   - Device fingerprinting (browser fingerprint hash stored at registration)
   - Rate limiting on account creation endpoints
   - Flagging (not blocking) potential duplicates for manual review to avoid false positives
   - Privacy-preserving approach: fingerprint data must not collect more data than necessary (FR-013 acceptance criteria)

4. **Consent Flow Integration**: The consent flow must be implemented as a mandatory post-authentication step. Users cannot access the tool until consent is recorded. Consent records must be versioned and auditable. Re-consent is required when consent terms change.

5. **Data Residency Consideration**: Supabase's closest data centre to the UK is EU (Frankfurt). The DPO must assess whether this satisfies ICO guidance on international transfers of special category health data. If UK-soil residency is required, the exit strategy is self-hosting the open-source Supabase stack on AWS eu-west-2 (London).

6. **MFA Rollout**: MFA (TOTP) is optional at launch per NFR-SEC-001. Consider making MFA mandatory for Clinical Reviewer and Admin roles from Day 1, and encouraging (but not requiring) MFA for users via progressive disclosure during onboarding.
